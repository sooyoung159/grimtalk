# LIVE_VALIDATION_CHECKLIST

Kanana API 일일 10회 제한 기준, **live는 검증용**으로만 사용한다.

- 모드 전환: `.env.local`에서 `NEXT_PUBLIC_KANANA_MODE=live`
- 리허설은 항상 `mock`으로 먼저 진행

---

## A) live 테스트 전 준비

- [ ] mock 모드에서 전체 흐름 1회 확인 (랜딩→결과)
- [ ] `.env.local`이 `NEXT_PUBLIC_KANANA_MODE=live`인지 확인
- [ ] 카메라/마이크 권한 허용 상태 확인
- [ ] 테스트 그림 3장 준비 (이름/놀이/위로)
- [ ] 테스트 문장 사전 메모
  - 이름형: “안녕! 너 이름이 뭐야?”
  - 놀이형: “같이 놀자!”
  - 위로형: “조금 무서워. 같이 있어줄래?”

---

## B) 1회 테스트 때 반드시 확인할 항목

- [ ] 녹음 시작/종료가 정상 동작하는가
- [ ] 서버 응답이 타임아웃 없이 돌아오는가
- [ ] assistantText가 그림/발화를 반영하는가
- [ ] 오디오 재생이 되는가 (또는 미제공 시 fallback이 자연스러운가)
- [ ] 캐릭터 카드(이름/정체/질문)가 어색하지 않은가
- [ ] 방금 대화 영역이 최근 맥락을 올바르게 보여주는가

---

## C) 권장 테스트 순서 (호출 예산 절약)

### 1차 (핵심 성공 확보)
1. 이름 묻기형 (기본 성공 시나리오)
2. 같이 놀자형 (상호작용/에너지 확인)
3. 위로형 (감정 반응 확인)

### 2차 (필요 시)
4. 프롬프트 수정 반영 확인 1회
5. 회귀 체크 1회 (가장 잘 나온 시나리오 재실행)

### 3차 (문제 있을 때만)
6~8. 오디오/응답 이슈 재현용
9~10. 버퍼 (데모 직전 최종 확인용)

---

## D) 실패 시 기록 템플릿

아래 템플릿을 테스트 1회마다 짧게 기록한다.

```md
- 날짜:
- 모드: live
- 그림 종류:
- 발화:
- 실제 응답(요약):
- 오디오 재생: O / X
- 만족도(1~5):
- 다음 수정 포인트:
```

---

## E) 호출 예산 운영 가이드 (10회/일)

- **3회**: 핵심 시나리오 검증 (이름/놀이/위로)
- **3회**: 프롬프트/회귀 체크
- **2회**: 오디오/재생 이슈 재현
- **2회**: 데모 직전 버퍼

원칙:
- mock에서 확신이 들지 않으면 live 호출하지 않는다.
- 같은 조건 재시도는 최대 2회까지만 하고 기록을 남긴다.
- 데모 당일에는 최소 2회 이상 버퍼를 남긴다.

---

## 빠른 실행 요약 (데모 직전 30초)

1. `.env.local` → `NEXT_PUBLIC_KANANA_MODE=live`
2. 이름형 1회로 상태 확인
3. 이상 없으면 놀이형, 위로형 순으로 진행
4. 문제 생기면 즉시 mock로 전환해 UI 시연 계속
