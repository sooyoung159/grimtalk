# KANANA_AUDIO_STRATEGY

## 1) 현재 문제 요약

관찰된 상태:
- request shape 자체는 upstream으로 정상 전달됨 (`text` / `image_url` / `input_audio`)
- upstream 응답은 `400` + `{"detail":"cannot load audio file."}`

해석:
- 스키마 문제가 아니라 **오디오 디코딩/파싱 실패** 가능성이 높음
- 특히 모바일/브라우저 기본 녹음 포맷(`audio/webm;codecs=opus`)을 Kanana가 읽지 못할 수 있음

---

## 2) 왜 webm/opus가 실패 후보인지

- 에러 메시지가 shape mismatch보다 파일 로드 실패(`cannot load audio file`)에 가까움
- 모바일 브라우저에서 흔한 출력은 `webm/opus`, `mp4/m4a`, `aac` 등으로 분산됨
- 모델/엔진에 따라 컨테이너(webm/mp4) + 코덱(opus/aac) 조합 지원 범위가 다름
- Kanana 예시에서 `input_audio.format="wav"`가 제시되어 있어, wav 경로가 더 안전한 가능성

---

## 3) 목표 포맷을 wav로 보는 이유

- 제공된 예시에서 `format: "wav"`가 명시됨
- PCM 기반 WAV는 디코딩 호환성이 상대적으로 높음
- 디버깅 시 코덱/컨테이너 변수(webm/opus, mp4/aac)를 줄일 수 있음
- 실패 원인을 스키마/인코딩/데이터 손상 중 어디인지 분리하기 쉬움

---

## 4) 가능한 구현 전략

## 전략 A) 서버(ffmpeg) 기반 변환 (권장)

흐름:
1. 브라우저에서 원본 업로드(webm/mp4/m4a/aac)
2. `route.ts` 또는 별도 유틸에서 ffmpeg로 wav(예: mono, 16kHz, pcm_s16le) 변환
3. 변환된 wav base64를 `input_audio.data`로 전송, `format: "wav"`

장점:
- 클라이언트 환경 차이를 서버에서 흡수
- 모바일 브라우저별 MIME 차이를 단일 파이프라인으로 통일
- Kanana 입력 포맷을 강제할 수 있어 재현성 좋음

단점:
- ffmpeg 런타임 의존성 추가
- 서버 CPU/응답시간 증가 가능
- 배포 환경(서버리스 포함)에서 바이너리/실행 정책 점검 필요

---

## 전략 B) 클라이언트에서 wav 생성 후 업로드

흐름:
1. 브라우저에서 녹음 후 Web Audio API/라이브러리로 wav 인코딩
2. 서버는 변환 없이 wav 전달

장점:
- 서버 부하 감소
- 서버 구현 간단

단점:
- 브라우저별 오디오 파이프라인 차이로 구현 복잡
- 모바일 성능/메모리 부담 가능
- 클라이언트 코드가 무거워질 수 있음

---

## 전략 C) 임시로 wav 업로드만 허용

흐름:
1. webm/mp4 업로드 차단
2. wav 녹음 가능한 환경만 허용

장점:
- 가장 빠른 리스크 차단
- 서버 구현 최소

단점:
- 모바일 호환성 크게 저하
- 사용자 경험 악화(많은 기기에서 실패)

---

## 5) Next.js 로컬 개발환경 기준 추천

**추천: 전략 A (서버 ffmpeg 변환)**

이유:
- 현재 이미 서버 라우트에서 업로드 처리 중이라 삽입 지점이 명확함
- 원인(코덱/컨테이너 불일치) 제거 효과가 가장 큼
- 추후 운영에서도 “입력 다양성 → 서버 표준화(wav)” 패턴이 유지보수에 유리

권장 변환 목표(초안):
- 컨테이너: wav
- 코덱: pcm_s16le
- 채널: mono
- 샘플레이트: 16000(또는 Kanana 권장값 확인 후 조정)

---

## 6) 방법별 장단점 요약 표

| 방법 | 장점 | 단점 | 권장도 |
|---|---|---|---|
| A. 서버 ffmpeg 변환 | 호환성/재현성 높음, 모바일 차이 흡수 | 런타임 의존성/CPU 비용 | 높음 |
| B. 클라이언트 wav 생성 | 서버 단순화 | 브라우저 편차/구현 복잡도 | 중간 |
| C. wav만 허용 | 즉시 단순화 | 실사용 호환성 낮음 | 낮음(임시) |

---

## 7) 다음 단계(구현 전 체크)

1. Kanana가 권장하는 wav 사양(샘플레이트/채널/비트뎁스) 확인
2. 로컬에서 ffmpeg 접근성 확인
3. 변환 전/후 파일 정보(mime, byteSize, duration) 디버그 로그 설계
4. 변환 실패 시 사용자 메시지/재시도 정책 정의
